Version Control

4. Git, GitHub, AI 협업 이해하기

바이브코딩에서는 AI Agent가 파일을 만들고 수정하지만, 그 변경을 안전하게 남기고 공유하는 기준은 Git과 GitHub가 담당한다. 이 문서는 비개발자가 Git, GitHub, AI의 역할을 구분하고 프로젝트 진행 중 무엇을 확인해야 하는지 설명한다.

세 가지 역할

Git은 내 컴퓨터 안에서 프로젝트 변경 이력을 기록하는 도구다. GitHub는 그 기록을 인터넷 저장소에 올려 팀과 공유하고, 배포 서비스와 연결할 수 있게 하는 플랫폼이다. AI Agent는 코드를 대신 작성하거나 수정 제안을 만들 수 있지만, 어떤 변경을 저장하고 공유할지는 사람이 확인해야 한다.

Git과 GitHub 차이

구분 쉽게 말하면 비개발자가 확인할 것
Git 프로젝트 파일의 변경 기록을 남기는 도구다. 어떤 파일이 바뀌었는지, 저장 메시지가 알아보기 쉬운지 확인한다.
GitHub Git 기록을 온라인에 올려 보관하고 공유하는 서비스다. 저장소 이름, 공개 여부, 팀 접근 권한, 배포 연결 상태를 확인한다.
Commit 특정 시점의 변경 내용을 저장하는 단위다. "로그인 화면 추가", "비용 문서 수정"처럼 목적이 보이는 메시지인지 확인한다.
Branch 기존 작업을 망가뜨리지 않고 별도 흐름에서 수정하는 작업 공간이다. 실험 작업, 수정 작업, 배포 준비 작업이 main과 분리되어 있는지 확인한다.
Pull Request 변경 내용을 main에 합치기 전에 설명하고 검토하는 요청이다. 무엇을 바꿨는지, 화면 확인 결과가 있는지, 위험한 변경은 없는지 본다.

AI와 Git의 관계

AI Agent는 프로젝트 파일을 빠르게 바꿀 수 있기 때문에 Git이 더 중요해진다. Git 기록이 없으면 AI가 만든 변경을 비교하거나, 문제가 생겼을 때 이전 상태로 돌아가기 어렵다.

AI가 코드를 만들었다고 바로 저장하거나 배포하지 않는다. 먼저 어떤 파일이 바뀌었는지 보고, 변경 목적이 맞는지 확인한 뒤 커밋한다.

프로젝트에서 권장 흐름

  1. 저장소를 먼저 만든다. 프로젝트 이름, 설명, 공개 여부를 정하고 GitHub 저장소를 준비한다.
  2. 작업 단위를 작게 나눈다. "전체 앱 만들기"보다 "로그인 화면 만들기", "비용 문서 추가하기"처럼 작은 단위로 요청한다.
  3. AI가 바꾼 파일을 확인한다. 변경된 파일 목록과 내용 차이를 보고, 의도하지 않은 파일이 바뀌지 않았는지 확인한다.
  4. 의미 있는 커밋 메시지를 남긴다. 나중에 봐도 어떤 작업인지 알 수 있게 작성한다.
  5. GitHub에 올린 뒤 화면이나 배포를 확인한다. GitHub Pages, Vercel, Netlify 같은 배포 서비스와 연결되어 있다면 push 후 결과 화면을 확인한다.

비개발자가 확인할 체크리스트

  • GitHub 저장소가 공개 저장소인지 비공개 저장소인지 확인했는가?
  • 비밀번호, API 키, 개인정보 같은 민감한 값이 파일에 들어가지 않았는가?
  • AI가 수정한 파일 목록을 확인했는가?
  • 커밋 메시지만 봐도 변경 목적을 알 수 있는가?
  • 배포와 연결된 브랜치가 무엇인지 알고 있는가?
  • 문제가 생겼을 때 이전 커밋으로 돌아갈 수 있도록 작은 단위로 저장했는가?
  • 팀원이 있다면 Pull Request로 검토할 흐름을 정했는가?
