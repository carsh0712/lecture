Database Decision

3. 데이터베이스 결정하기

데이터베이스는 프로젝트가 사용하는 데이터를 저장하고 다시 꺼내 쓰는 공간이다. 모든 프로젝트에 데이터베이스가 필요한 것은 아니므로, 먼저 저장할 데이터가 있는지부터 판단한다.

간단한 판단 흐름

flowchart TD

  B["DB 없음

  C{"데이터를 사람이

주요 선택지

선택지 쉽게 말하면 어울리는 경우 주의할 점
DB 없음 저장할 데이터 없이 화면이나 문서만 제공한다. 매뉴얼, 소개 페이지, 고정된 안내 자료 사용자가 입력한 내용을 저장하거나 조회할 수 없다.
파일 CSV, JSON, 엑셀 같은 파일에 데이터를 보관한다. 작은 목록, 설정값, 사람이 직접 관리해도 되는 데이터 여러 사람이 동시에 수정하면 충돌이 생기기 쉽다.
SQLite 하나의 파일처럼 관리되는 가벼운 데이터베이스다. 개인용 도구, 작은 내부 앱, 초기 MVP 동시 사용자가 많거나 서버 여러 대에서 운영하는 앱에는 조심해야 한다.
PostgreSQL 운영용 웹앱에서 많이 쓰는 안정적인 데이터베이스다. 여러 사용자가 데이터를 입력, 수정, 검색하는 업무 도구 백업, 권한, 접속 정보, 비용 관리가 필요하다.

처음에는 이렇게 고른다

  • 안내 페이지나 문서라면 DB 없음으로 시작한다.
  • 데이터가 작고 사람이 파일로 관리할 수 있으면 CSV나 JSON 파일을 검토한다.
  • 작은 앱에서 데이터를 저장해야 하지만 사용자가 많지 않다면 SQLite를 검토한다.
  • 여러 사람이 동시에 쓰거나 운영 데이터가 중요하면 PostgreSQL을 검토한다.
  • 로그인, 권한, 파일 저장까지 필요하면 Supabase 같은 관리형 서비스를 함께 검토한다.

비개발자가 확인할 질문

  • 사용자가 입력한 내용을 나중에 다시 봐야 하는가?
  • 데이터를 검색하거나 필터링해야 하는가?
  • 여러 사람이 동시에 수정할 가능성이 있는가?
  • 삭제되면 안 되는 중요한 데이터인가?
  • 백업이 필요한가?
  • 개인정보나 회사 내부 정보가 들어가는가?
데이터가 중요할수록 "일단 파일로 저장하자"는 선택이 위험해질 수 있다. 운영에 쓰는 데이터라면 처음부터 백업과 권한을 함께 생각해야 한다.

AI Agent에게 말하는 예시

우리 프로젝트에 데이터베이스가 필요한지 판단해줘.

상황:
- 사용자가 문의 목록을 등록하고 상태를 수정한다.
- 사내 직원 5명이 함께 사용한다.
- 데이터는 나중에 검색해야 한다.
- 삭제되면 곤란하므로 백업이 필요하다.

비교해줄 항목:
- DB 없음, 파일, SQLite, PostgreSQL 중 어떤 선택이 맞는지
- 처음 MVP에서 추천하는 저장 방식
- 나중에 운영용으로 바꿔야 할 가능성
- 개인정보나 권한 관리에서 주의할 점
- 비개발자가 확인해야 할 데이터 항목
